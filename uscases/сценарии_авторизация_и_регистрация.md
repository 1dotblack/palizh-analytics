# Сценарии: авторизация и регистрация

## Назначение

Документ фиксирует единый контур онбординга и авторизации для B2B-платформы:

- от заявки "стать клиентом";
- до активации в `1С`;
- до входа пользователей компании в ЛК.

## Границы

- `MVP`: регистрация юрлица через заявку, ручная проверка/активация в `1С`, создание доступа в ЛК, вход, восстановление пароля.
- `Post-MVP`: саморегистрация с автоодобрением, расширенные механики подтверждения и MFA.

## Use cases (MVP)

| ID | Предусловия | Основной поток | Альтернативы/ошибки | Результат | Источник данных | Уведомления | Критерии приемки (QA) |
| --- | --- | --- | --- | --- | --- | --- | --- |
| AUTH-01 | Пользователь не авторизован | Заполняет форму "Стать клиентом" (ИНН/КПП/email/телефон) -> отправляет заявку | Невалидные реквизиты/дубликат | Заявка принята в обработку | Платформа | Email о принятии заявки | Создается заявка с уникальным ID |
| AUTH-02 | Заявка создана | Менеджер/оператор обрабатывает заявку в контуре `1С` -> активирует контрагента | Отказ/запрос доп.данных | Контрагент активирован или отклонен | `1С` + ручной процесс | Email о результате проверки | Статус заявки синхронизирован |
| AUTH-03 | Контрагент активирован | Создается пользователь ЛК (первичный доступ) -> пользователь выполняет вход | Неверный пароль/неактивный аккаунт | Успешный вход в ЛК | Платформа + `1С` (флаг активности) | Email с доступом/инструкцией | Вход доступен только активному клиенту |
| AUTH-04 | Пользователь имеет доступ | Запрашивает восстановление пароля -> получает ссылку -> задает новый пароль | Просроченная ссылка/ошибка токена | Пароль обновлен | Платформа | Email восстановления | После смены пароля старые токены недействительны |
| AUTH-05 | Есть повторная заявка | Система/оператор проверяет дубли по ИНН/email/GUID | Конфликт дублей | Принято решение: связать/отклонить/обновить | Платформа + `1С` | Email о статусе | Не создаются неконтролируемые дубликаты |

## Экраны для дизайна

Перечень **экранов и состояний UI** по шагам `AUTH-*` (для макетов; детальные шаги — в потоках).

| ID | Экраны и состояния |
| --- | --- |
| AUTH-01 | Точка входа (CTA на витрине) → форма «Стать клиентом» → валидация полей → успех «Заявка принята»; опционально модал/экран при дубликате заявки |
| AUTH-02 | В основном **вне UI клиента** (`1С` / админка / почта); при необходимости в клиенте — состояние «ожидание активации»; макеты писем — отдельно |
| AUTH-03 | Страница входа → ошибки входа → **попытка входа до активации** (отдельное сообщение) → после успеха: корень **ЛК** / онбординг пустого состояния |
| AUTH-04 | Запрос восстановления пароля → «Письмо отправлено» → форма нового пароля по ссылке → успех; просроченный/невалидный токен → повторный запрос |
| AUTH-05 | Экран или модал конфликта дубликата (связать / отклонить / обновить — по финальному UX); согласовать с блоком «Негативные сценарии (MVP)» ниже (уже активный контрагент) |

Сквозные шаги с нумерацией экранов: [потоки_для_дизайна/поток_01_регистрация_активация_первый_вход.md](потоки_для_дизайна/поток_01_регистрация_активация_первый_вход.md), [потоки_для_дизайна/поток_02_вход_восстановление_пароля.md](потоки_для_дизайна/поток_02_вход_восстановление_пароля.md).

## Негативные сценарии (MVP)

- Попытка входа до активации в `1С` -> доступ запрещен, понятное сообщение.
- Попытка регистрации с уже активным контрагентом -> переход в восстановление/контакт с менеджером.
- Повторное использование ссылки восстановления -> отклонение с предложением запросить новую.
