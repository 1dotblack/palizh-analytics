# Саммари: демо прототипов платформы

**Дата:** 2026-04-15  
**Источник:** `Интервью и встречи/Транскрайб/2026-04-15_Демо_прототипов.txt`  
**Стиль:** нейтрально-деловой, акцент на решениях и доработках

**Контекст:** демонстрация прототипов заказчику (Palizh) и синхронизация понимания перед дизайн-концепцией. На связи были представители Palizh и сторона разработки/прототипирования (в т.ч. Александр — ведущий демо, Настя, Стас по 1С, Лера и др.). Во встрече были перебои со звуком; часть тезисов повторяли.

---

## 1. Подход к прототипам (как делали)

- Прототипирование **по путям пользователя** (последовательность экранов, контекст входа в шаг, линейность), чтобы не терять связность и заранее видеть блокирующие места.
- Ориентир на **содержание и формулировки**, а не «квадратики»: готовность зайти в **дизайн-концепцию** с проработанными смыслами и текстами.
- Охвачено **около 7–8 потоков**. Отдельные **служебные** экраны (создание/редактирование пользователей и т.п.) сознательно **не делали** — их проще собрать статично позже.
- **Важно:** прототип — не финальный визуал; оценка «эстетики» и каруселей отложена на этап дизайна; на демо — логика блоков и сценарии.

## 2. Потоки и ключевые договорённости по экранам

### 2.1 Главная (незарегистрированный пользователь)

| Тема | Решение / смысл |
|------|------------------|
| Назначение | Донести, что это **B2B-кабинет**, ценности: доставка, простота заказа, зачем регистрироваться. |
| Баннер сверху | При **входе/регистрации** баннер про ЛК **скрывается**; остаются контентные баннеры (акции, новинки и т.д.). |
| Карусель | Идея **карусели** для баннеров **не исключена**; на прототипе может быть статика — на дизайне уточнить. |

### 2.2 Регистрация и первый вход

| Тема | Решение / смысл |
|------|------------------|
| ИНН | Ввод ИНН (**юрлицо или ИП**), **поиск/валидация**, подтягивание **названия, КПП и пр.** с **внешнего сервиса** (не ФНС в текущей постановке); пользователь **не редактирует** подтянутые реквизиты вручную. |
| Название компании | Отдельно не заставляют вводить, если всё приходит по ИНН — для Palizh **упрощает** обмен с 1С. |
| Ошибки | Учтены состояния ошибок; **длина ИНН** — 12 или 10/14 для разных типов; явная ошибка, если не 12 и не 14 (и не буквы). Запрос к сервису при **корректном формате**. |
| Далее | Заявка принята → письмо для активации; экраны ошибок входа; первый вход в ЛК. |

### 2.3 Восстановление пароля

- Стандартный поток: запрос ссылки, письмо, сброс, ошибки (неверные данные, **просроченная ссылка**), успех → вход.
- Особых продуктовых вопросов не поднято.

### 2.4 Каталог и карточка товара

| Тема | Решение / смысл |
|------|------------------|
| Поиск | Открытие подсказок, корректировки запроса, состояние «ничего не найдено». |
| Карточка | Описание, характеристики, превью, **наличие и цена** для активированного пользователя; тост «добавлено в корзину»; **нет в наличии** — ориентир по поступлению; **гость** — сообщение о необходимости авторизации. |
| Номенклатура 1С | **Финальная выгрузка ещё в работе** (Стас); экран карточки/группировки **уточнят после реальных данных**. |
| Цвет / варианты (Palizh) | **~95%** позиций — **отдельные номенклатурные позиции** (например, паста зелёная / красная как разные товары); **~5%** (миксы) — где уместен **переключатель по цвету** (цветов много, SKU мало). |
| Нестандартная заявка | Отдельная кнопка **«Оформить нестандартную заявку»** → **модалка по клику**, не по таймеру/авто. План: виджет **чата с менеджером**, вложения файлов. Уточнено: не всплывает «само» при заказе — только по действию пользователя (обсуждалась путаница с формулировками). |

**Фильтры (каталог):** задуманы как **динамические** по атрибутам товаров; переключатель **«промышленность»** — предварительно, детали **через сегменты** (Наташа — отдельно с Александром).

### 2.5 Корзина, доставка, оформление

| Тема | Решение / смысл |
|------|------------------|
| Корзина | Мини-карточки: название, артикул, фасовка, количество, избранное, скидка/цена, удаление. |
| «Скидка» в UI | **Не показывать скидку как отдельный акцент**, если у части клиентов **наценка** по соглашению (удалённые районы и т.д.). |
| Две цены | Предпочтительно показывать **цену за некратную упаковку** и **за кратную (коробку)** — нагляднее, чем абстрактный %; процент скидки можно **не выводить**, если понятно из двух цен. |
| Данные из 1С | Стас передаёт **две цены** + параметры кратности; логику расчёта **не дублировать на платформе** — отображать то, что пришло. |
| Бонусы в корзине | **Убрать** начисление бонусов из корзины: бонусы **месяц/квартал/объём**, не «с заказа сразу»; позже — **отдельный раздел** ЛК (с бухгалтерией). |
| Прайс скачать/загрузить | Блок **оставлен как напоминание**, в первую очередь **не реализуют** — сначала полный процесс заказа; вернуться отдельно. |
| Порог 120 тыс. | Если сумма **ниже** — показать, **сколько не хватает** до бесплатной доставки; **переход к оформлению не запрещать** (срочность, ТК и т.д.). При оформлении — **явно уточнять способ доставки** (своя / ТК). |
| Адрес доставки | Хранить **на портале**: выбор ранее введённых + «новый адрес»; **вместе с заказом передавать** в 1С. Опционально в будущем: подстановка из 1С, если в партнёре/контрагенте заполнено (обсуждалось без жёсткой фиксации на демо). |
| ТК | Обсуждались **обязательные поля** и **режим работы точки** (данные в контрагенте/партнёре); синхронизация с маршрутами — «брать режим работы из партнёра» при корректном заполнении в 1С. |

**Открытый сложный сценарий:** заказ **< 120 тыс.**, но клиент **встречает машину на трассе** / забирает у водителя — **бесплатно без довоза до адреса**. Это **ломает простую логику** «120 тыс. = бесплатно до двери». Договорённость: **отдельный короткий созвон** (7–15 мин); на стороне прототипа — идея **третьего варианта** («заберу сам» / терминалы Москва, Краснодар, Новосиб и т.д. + комментарий для водителя). Обсуждалось: не зашивать жёстко 120 тыс. только на фронте — возможен **признак из 1С** (сложность: до оформления заказа контрагент уже известен — требует отдельной проработки).

### 2.6 Подтверждение заказа и история

- Экран подтверждения с введёнными данными; **ближайшая дата** (ориентир) — надежда на данные из 1С.
- **История заказов:** фильтры/вкладки (новые, завершённые, отменённые); **нестандартные заявки** в общем списке с **тегами/типом**; статусы визуализированы широко для обсуждения.
- **Отмена заказа:** только в статусе **«обрабатывается»** (до того как менеджер/1С перевели дальше); после перехода в доставку — **нельзя** (согласовано с ожиданием «как только взяли в работу — клиент не ломает»).

### 2.7 Карточка заказа и документы

| Тема | Решение / смысл |
|------|------------------|
| Документы | Подгружаются **по мере прохождения цепочки** из 1С (счёт, УПД и т.д. — как печатные формы/образцы на прототипе). |
| Бонусы | Убрать с экрана заказа (как в корзине). |
| Состав заказа (B2B) | Предпочтение Palizh: **компактная таблица** без крупных превью — название, количество, сумма; желательно видеть **упаковки** (сколько коробок из штук), чтобы не считать вручную. |

**Повтор заказа:** вся логика через **наполнение корзины**; экран с **ошибками добавления** — часть позиций не попала (нет на складе / снято с производства). **Уточнённая логика:** **нет на складе** — можно **всё равно вести в корзину** (под производство); **снято с производства** — вниз списка / не оформлять или явно исключить; **обязательна возможность продолжить** оформление при частичных проблемах. Зафиксировать в ТЗ окончательно после согласования формулировок.

### 2.8 Документы и акт сверки

| Тема | Решение / смысл |
|------|------------------|
| Запрос акта сверки | По кнопке — период; **не «письмо менеджеру» как основной путь**, а **триггер в 1С** → формирование и выдача клиенту (исключить менеджера из рутины). В прототипе поправить формулировки/логику под это. |
| Нумерация в UI | Уточнение: фраза «заказ №…» у запроса акта — **скорее ошибка прототипа**; акт сверки **за период**, не привязка к одному заказу. |
| Раздел «Документы» | Много типов документов (к заказам и к контрагенту в целом: договор, доп. соглашения); **поиск/фильтр** по дате, **типу**, **номеру**; переход из найденного докума к **карточке заказа** и наоборот — сквозная навигация. |
| Админка | Настройка **куда слать** уведомления по типам заявок (в т.ч. акты) — обсуждалось в контексте маршрутизации. |

### 2.9 Не успели на встрече

- **Претензии**, **нестандарт** (детальная последовательность), **обучение** — попросили **просмотреть самостоятельно**; при необходимости вынести вопросы списком.
- **Табличный «каталог заказа» в ЛК** (референс TCoffee): отдельно от витрины с карточками — быстрый набор **без лишних кликов**; аргумент — разные задачи «изучить ассортимент» vs «быстро набить заказ» (связь с недавним интервью клиента). **Решение не принято** — «подумаем, потом скажем»; возможное **дублирование** витрины vs удобство.

## 3. Интеграция 1С (Стас)

- Обмен **номенклатуры** в разработке/обкатке; после «отмашки» о стабильном коннекте — дальше **остатки, цены, заказы** и т.д.
- На демо: **не вся** номенклатура с ценами/остатками — **отдельные сущности**, отдельные выгрузки.
- Стас: обмен **работает**; просьба со стороны платформы **проверить лимиты/offset’ы** и стабильность при повторных запросах.
