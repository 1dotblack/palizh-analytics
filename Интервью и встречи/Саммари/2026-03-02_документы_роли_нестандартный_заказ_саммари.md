## 1. Документооборот

### 1.1 Начало работы с новым клиентом (до договора)

| Элемент | Описание |
|--------|----------|
| **Коммерческое предложение (КП)** | Формирует региональный менеджер. Конкретной формы нет: при наличии реквизитов могут отправить в виде счёта или просто описать условия в письме (email). Не формируется в отдельной системе — письмо, Excel или PDF. |
| **От клиента запрашивают** | Карта партнёра и документы для оформления договора и карточки контрагента. |

### 1.2 Договор

| Вопрос | Ответ |
|--------|--------|
| С какой суммы заключают договор? | Договор — от 120 тыс. руб. (первый заказ). При сумме меньше 120 тыс. — заведение в программе «строчкой по счёту», отгрузка по 100% предоплате, без отдельного договора по ставке. |
| Кто формирует договор? | Менеджер по сопровождению. Есть несколько видов шаблонов. |
| Редактирование договора клиентом? | Не разрешают редактировать шаблон. При возражениях клиент оформляет **протокол разногласий**. |
| Стандартный vs нестандартный договор | Чаще подписывают стандартные; протоколы разногласий бывают, но реже. При нестандартном: протокол разногласий → задача в **MasProject** → юрист согласовывает, при необходимости подключает ПЭО, отдел качества и др. |
| Куда заносят подписанный договор? | Сначала в 1С заводится «строчка» контрагента. После подписания в 1С вкладывают подписанный скан или документ через ЭДО; юрист отмечает в программе. |

### 1.3 ЭДО (электронный документооборот)

- Сейчас не у всех клиентов; ориентировочно 50/50. Планы — переводить всех на ЭДО; сроки зависят от законодательства (пока жёстко не закреплено между контрагентами).
- Менеджер по сопровождению получает задачу по подключению клиентов к ЭДО; регулярно (раз в 1–2 месяца) делают срезы и активно подключают.

### 1.4 Акт сверки

| Вопрос | Ответ |
|--------|--------|
| Кто обрабатывает запрос? | Менеджеры по сопровождению. |
| Частота запросов | Обычно раз в квартал (квартальные и годовые). Реже — за прошедший месяц. |
| Процесс | Клиент запрашивает (например, по email) → формируют акт в 1С → отправляют по email или через бухгалтерию по ЭДО. Подписание в ЭДО — бухгалтер; менеджеры только просматривают. |

### 1.5 УПД и документы после отгрузки

- **УПД** создаётся кладовщиками в момент списания товара (расходный ордер), когда формируется отправка (машина/ТК). При необходимости добавляют ТТН.
- **Вопрос по ЛК:** хранить документы на платформе или по запросу «Сформировать УПД» дергать 1С и отдавать документ без хранения? Предпочтение клиента (Андрей): **без хранения** — по нажатию кнопки запрос в 1С, 1С отдаёт УПД, чтобы «сервера не распухали».

### 1.6 Оплата и контроль

- Оплаты: бухгалтерия выгружает банк в 1С (документ «Безналичные платежи»). По наименованию клиента и сумме проверяют поступление денег.
- **Важно:** отгрузку в 1С система не даёт провести, если по договору предусмотрена предоплата и деньги не пришли. При постоплате отгрузка возможна без проверки оплаты.
- Контроль клиентом своей задолженности — через менеджера (напоминания и т.п.).

### 1.7 Документы в ЛК для MVP

- Для MVP достаточно: **счёт, УПД, ТТН, паспорт качества, акт сверки**. Счёт-фактура = УПД; коммерческое предложение по смыслу = счёт.
- **После MVP** планируется «докрутить» механизм бонусов в ЛК — это бухгалтерский вопрос (озвучено в встрече).
- **История запросов/формирования документов** в ЛК (статусы «запрошен» / «сформирован») — не нужна. Достаточно хранить сами документы и давать возможность их скачать. История изменений по **заказу** (когда и что поменялось в составе) — нужна.

## 3. Статусы заказа (для платформы)

Обсуждена целевая цепочка статусов заказа в ЛК (привязка к событиям в 1С — детализировать позже):

| № | Статус | Комментарий |
|---|--------|-------------|
| 1 | Обрабатывается | Заказ заведён в ЛК |
| 2 | В производство / производится | Не хватает товара, создан заказ на производство |
| 3 | Готов к сборке | Реализация создана, остатки есть, но расходник ещё не проведён |
| 4 | Готов к отгрузке / машина готова к отправке | Склад провёл расходник, партия скомплектована |
| 5 | Отправлен / машина ушла | Далее — либо трек-номер (ТК), либо данные водителя (своя машина) |
| 6 | Завершён | Старые заказы помечать как завершённые |

- **Трек-номер** — только для доставки через ТК. Для своей машины — **контактные данные водителя** (ФИО, телефон, паспорт (серия/номер), автомобиль, прицеп при наличии, иногда водительское удостоверение). Данные нужны для связи и для пропускной системы на территории клиента.
- **Когда известен водитель:** в момент формирования задания на перевозку логистом (например, в пятницу — для отгрузки в пятницу; в четверг до 17:00 — заявки на пятницу, в пятницу возможны добавки по согласованию). Данные по водителю клиентам отправляет Ирина (логист).

## 4. Роли клиентов в ЛК

| Роль на стороне клиента | Кто взаимодействует с Palizh | Что должен видеть/делать в ЛК |
|-------------------------|------------------------------|--------------------------------|
| **Снабжение / закупщик** | Чаще всего; формирование заказа | Каталог, корзина, заказы, повтор заказа |
| **Бухгалтер** | Вопросы по счетам, актам сверки | Документы (в т.ч. акт сверки) |
| **Директор / собственник** | Иногда | Всё: заказы + документы |

- На стороне клиента бывает один контакт, а бывает несколько (снабжение, бухгалтер, технолог, директор) — по-разному.
- **Вывод по ролям в ЛК:** минимум три роли — (1) полный доступ (директор/собственник); (2) закупщик (заказы, каталог, корзина, повтор); (3) бухгалтер (документы, акты сверки). Разграничение по вкладкам/блокам (заказы, документы, профиль) достаточно; точечные права на блоки давать проще, чем более дробные настройки. Решение в встрече: одна роль (директор) видит **все вкладки**; остальные роли видят подмножества (например, закупщик — заказы + каталог, бухгалтер — документы); реализация через «роль 1 → вкладки 1,2,3; роль 2 → 2,3; роль 3 → только 1» проще, чем тонкие права на каждый блок.
- **Приглашение пользователей:** у компании клиента должен быть «мастер-аккаунт» (например, директор), который видит всё и может создавать под-пользователей с разными ролями (закупщики, бухгалтеры и т.д.) — «сдаём возможность самим создавать внутри пользователей».

## 5. Нестандартные заявки (услуги)

### 5.1 Типы

- **Колеровка по каталогу** — сборка цвета по каталогу.
- **Колеровка по образцу клиента** — клиент присылает материал (база) и образец (курьером); лаборатория подбирает цвет, делает пробник или рецептуру.
- **Оборудование** — продажа оборудования (в т.ч. дозаторов); доставка от поставщика, не со склада Palizh.
- **Обучение** — обсуждается отдельно.

### 5.2 Колеровка по образцу — процесс

1. Клиент обращается → запрашивают материал (база) и образец.
2. Клиент присылает курьером → передают в лабораторию.
3. В 1С оформляют как обычный заказ, номенклатура — «сборка цвета»; в комментарии — база, нужный цвет, вводные от клиента.
4. Создают документы: заказ покупателя, заказ на производство (участок — лаборатория).
5. Лаборатория выполняет: либо пробник, либо рецептура (в Telegram). Закрывают заказ в программе.
6. Дальше — как обычный заказ (реализация, расходник, отгрузка и т.д.).

Объём таких заявок: порядка 8 в месяц на всех менеджеров; в основном ведёт **региональный менеджер**; менеджер по сопровождению может завести заказ и передать на производство как исполнитель.

### 5.3 Оборудование

- Реализация и расходник проводятся у Palizh (транзит); логисту передают информацию; поставщик может отправить клиенту сам, тогда передают клиенту данные о сроках. Контроль этапов поставки ведётся.

### 5.4 Нестандартная заявка в ЛК

- Достаточно формата «свяжи меня с менеджером» с возможностью указать тип вопроса (колеровка, оборудование) и при необходимости прикрепить файл. Сильно автоматизировать не планируют — менеджер всё равно уточняет и заполняет в 1С.
- **Прикрепление файлов:** нужно. Примеры: Excel с составом заказа, паспорта безопасности, условия воспроизведения цвета. Форматы: PDF, Excel, Word.
- **Статусы нестандартной заявки:** урезанный набор (без «в цехе» и т.п.). Для колеровки/лаборатории — статус типа «в производстве/в работе»; для оборудования — цепочка как у обычного заказа, но без этапа «к производству». Для претензий — возможность прикреплять фото (бой, брак и т.д.).
- **Разработка цвета (Вера, B2C):** этап «разработка цвета» оформляется в 1С как заказ на производство с комментарием (участок лаборатории, «разработка цвета»); после него идёт обычный заказ на производство. В ЛК **отдельный статус «разработка цвета» не вводим** — это этап оказания услуги, а не статус документа; отображаем общий статус вроде «в работе» / «в производстве».

## 6. Решения и договорённости

- УПД в ЛК: по запросу из 1С, без хранения на платформе.
- MVP документов в ЛК: счёт, УПД, ТТН, паспорт качества, акт сверки; истории запросов документов не ведём. Бонусы в ЛК — после MVP, бухгалтерский вопрос.
- Статусы заказа — шесть этапов (обрабатывается → в производство → готов к сборке → готов к отгрузке → отправлен → завершён); привязка к 1С — уточнять технически.
- Роли в ЛК: минимум три (полный доступ, закупщик, бухгалтер); разграничение по вкладкам (директор — все вкладки, остальные — подмножества); мастер-аккаунт может создавать под-пользователей.
- Нестандартная заявка в ЛК: форма «связь с менеджером» + тип (колеровка/оборудование) + прикрепление файлов (PDF, Excel, Word). Этап «разработка цвета» (B2C) отдельным статусом в ЛК не показываем.
