## Саммари: Интервью 5 — 1С, обмен данными

**Дата встречи:** 13.03.2026  
**Источник:** транскрипт [`Palizh Интервью 5 13.03.26. 1С обмен данными.txt`](../Транскрайб/Palizh%20Интервью%205%2013.03.26.%201%D0%A1%20обмен%20данными.txt)  
**Тема:** процессы и механизмы обмена с 1С, документы, EDO, цены/скидки, онбординг клиентов и тестовый контур.

**Актуализация после интервью (согласование с командой заказчика):** для интеграции с платформой доступ к HTTP-сервисам 1С планируется **без VPN** — с **белого списка исходящих IP** окружений платформы (dev / stage / prod) на стороне заказчика. В тексте ниже может встречаться **VPN** как отражение **формулировок встречи 2026-03-13**; для проектирования брать **ЧТЗ 09** §4.1 и **`Техническая часть/Интеграция_1С.md`**.

## 1. Контекст и границы интервью

- Платформа **не предполагает личного кабинета менеджера**: менеджер общается с клиентом по email, телефону и через чат (виджет на сайте, у менеджера — единая точка входа).
- **Вся документация по заказу** (счета, УПД, ТТН, паспорта качества, акты сверки и т.д.) должна **получаться из 1С** и быть привязана к заказу.
- Интервью — установочная сессия: цель — зафиксировать, что уже есть в 1С, какие есть интеграции, какие механизмы обмена возможны и как подойти к проектированию API/шины.

## 2. Текущая система 1С и процессы

### 2.1 Конфигурация и версии

- Используется **1С ERP «Управление предприятием»**, версия **2.5**, релиз **2.5.22** (релиз длительной поддержки на текущий момент).
- Планируется новый релиз длительной поддержки ориентировочно в апреле; при этом «основы» конфигурации по идее не должны сильно поменяться, но полностью гарантировать это нельзя (типичная особенность 1С).

### 2.2 Какие процессы уже автоматизированы в 1С

- В 1С уже ведутся:
  - **Заказы клиентов** (создание, проведение).
  - **Счета** (формируются как печатные формы: PDF, Excel и т.п. на основе документа «Заказ клиента»).
  - **Отгрузки** (реализации, расходные ордера).
  - **Контрагенты, договоры, соглашения** (условия оплаты, отсрочка, типы цен).
  - **Правила доставки** (частично, через договоры/справочники; детально оформлены в распоряжениях ПЭО).
- Вне 1С по сути остаются:
  - **Подписание документов в ЭДО** (усиленные квалифицированные подписи, УКЭП).
  - Часть документооборота через бумажные носители/сканы для «староверов», которые принципиально не хотят переходить в ЭДО (но это временно — под давлением законодательства и «Честного знака» все будут вынуждены перейти).

### 2.3 Сервера и регламентные работы

- Есть регламентный **перезапуск сервера около 05:00**:
  - рестарт ОС/сервера;
  - поднимаются SQL, 1С и Apache.
- На время рестартов стоит предусмотреть:
  - толерантность интеграции к краткосрочной недоступности 1С;
  - возможные ретраи запросов с платформы.

## 3. Документы, ЭДО и хранение файлов

### 3.1 EDO и «Честный знак»

- Не все клиенты сейчас работают по ЭДО:
  - есть клиенты, которые **принципиально не хотят** ни отправлять, ни получать документы в ЭДО, предпочитая бумагу.
- **Тренд и законодательство:**
  - продукты Palizh (шпатлёвка) уже подпадают под **«Честный знак»**: передача УПД должна идти через ЭДО.
  - Планируется, что к концу **2027–2028** годов по остальным ЛКМ (лакокрасочные материалы) требования тоже усилятся.
  - Фактически, **все контрагенты будут вынуждены перейти к ЭДО**, иначе они не смогут торговать соответствующими товарами.
- Вывод: в перспективе стоит **как минимум оценить интеграцию платформы с ЭДО**, но на текущем этапе MVP можно ограничиться интеграцией с 1С и предусмотреть задел.

### 3.2 Хранение договоров и связанных документов

- В 1С есть справочники:
  - **Контрагенты**;
  - **Договоры**;
  - **Соглашения** (условия оплаты, этапы, отсрочка).
- К карточке **договора** в 1С можно **прикреплять файлы**:
  - собственно **договор**;
  - **дополнительные соглашения**;
  - и прочие вложения.
- Это означает:
  - платформа может получать **файлы договора и допсоглашений прямо из 1С** (как из карточки договора);
  - нет отдельной папки где-то «вне 1С» — связь «договор ↔ файлы» уже обеспечивается в 1С.

### 3.3 Форматы файлов и транспорт

- Документы в 1С формируются как **печатные формы** (PDF, Excel и др.).
- Для интеграции обсуждается:
  - передавать файлы в теле ответа как **base64** внутри JSON (с указанием типа/расширения);
  - платформа, получив JSON, смотрит на **расширение** и сохраняет файл с правильным типом.
- По сути:
  - **формат документа (PDF/Word/JPEG и т.п.)** — внутреннее дело 1С и бизнес-процесса;
  - для платформы главное — унифицированный способ получения: JSON + закодированный файл + метаданные (тип, имя, размер).

## 4. Механизм обмена: расписание, события, шина, очереди

### 4.1 Расписание vs события vs гибрид

- Варианты, обсуждённые на интервью:
  1. **Обмен по расписанию**:
     - платформа раз в N минут (например, каждые 5–10 мин) опрашивает 1С по API;
     - получает изменения (остатки, статусы, новые документы).
  2. **Обмен по событиям (вебхуки)**:
     - в 1С при значимых событиях (проведение расходника, создание маршрута, формирование документа) срабатывает обработчик;
     - 1С сама пушит изменения на платформу (или в шину), а платформа обновляет данные.
  3. **Гибридный вариант (предпочтительный):**
     - всё, что **некритично к задержке** — по расписанию (платформа «ходит» в 1С по API, например, для фонового обновления каталога/остатков);
     - всё, что **важно держать максимально синхронизированным** — через вебхуки/шину (статусы заказа, ключевые документы, события доставки).

### 4.2 Инициаторы обмена

- При гибриде **инициатором могут быть обе системы**:
  - **Платформа → 1С**:
    - создаёт заказ клиента;
    - запрашивает конкретный документ (счёт, УПД и т.п.);
    - запрашивает актуальные остатки/сроки по конкретным товарам.
  - **1С → Платформа**:
    - пушит событие о смене статуса (проведена реализация, создан маршрут, проведён расходный ордер и т.п.);
    - пушит событие о формировании документа (например, акт сверки готов).

### 4.3 Шина и очереди (RabbitMQ и др.)

- Обсуждался сценарий использования **общей шины / очереди сообщений** (например, RabbitMQ), чтобы:
  - сообщения (вебхуки) **не терялись** при кратковременных сбоях связи;
  - обеспечить **масштабирование** и надёжность обмена;
  - отделить приём событий от их обработки (производитель/потребители).
- В такой архитектуре:
  - 1С публикует сообщение (или фронт/бэкенд платформы проксирует событие) в очередь;
  - специализированный интеграционный сервис читает очередь и обновляет данные в БД платформы;
  - при этом ответы 200/400/500 с одной стороны не приводят к потере события (оно останется в очереди до успешной обработки).

## 5. Данные и бизнес-логика: цены, скидки, сроки, доставка

### 5.1 Цены, виды цен и персональные скидки

- В 1С цены организованы через:
  - **виды цен** (розница, опт, ВЭД и т.п.);
  - **соглашения**: для каждого контрагента в соглашении указан **вид цены по умолчанию** и могут быть заданы **персональные скидки в процентах**.
- При расчёте цены в 1С:
  - у контрагента есть базовый вид цены (например, «Оптовая»);
  - на него могут действовать **автоматические скидки**, прописанные в соглашении;
  - дополнительно менеджер может назначать **ручные скидки** (до ~18% сверх автоматических) на стороне 1С при работе с заказом.
- На интервью обсуждались два подхода для платформы:
  1. **Получать из 1С полностью пересчитанные «готовые» цены под каждого контрагента** (уникальная итоговая цена по каждому товару).
  2. **Получать из 1С базовые цены по виду цены + процент персональной скидки для контрагента** и применять скидку уже на стороне платформы.
- От подхода № 1 **отказались**:
  - хранить и обновлять отдельный прайс под каждого клиента неудобно и тяжело для витрины/кэша;
  - усложняется как интеграция, так и сопровождение каталога.
- Согласован подход № 2:
  - платформа:
    - получает из 1С:
      - **виды цен** (в т.ч. розничный — для витрины гостя, и оптовый/иной — для B2B);
      - **процент персональной скидки** по контрагенту/соглашению;
    - **отображает базовую цену** по соответствующему виду цены;
    - **применяет персональную скидку в корзине** ко всем товарам клиента (и может явно показывать величину скидки относительно базового вида цены).
  - ручные/разовые скидки менеджера в 1С остаются **только в 1С**: они не закладываются в модель витрины и отражаются уже в итоговых документах, а не в расчёте базовых цен на сайте.

### 5.2 Остатки и сроки производства

- Остатки по номенклатурам ведутся в 1С; по ним платформа будет получать:
  - текущие остатки;
  - **сроки производства / ожидаемую дату** при отсутствии товара.
- Сейчас:
  - если товара нет, менеджер оформляет **заказ на производство**;
  - в нём есть **плановая дата производства**;
  - при необходимости процессы внутри компании будут перестроены так, чтобы эта дата была актуальной и могла передаваться на платформу.
- Для платформы:
  - в карточке товара/корзине можно отображать «ожидаемую дату» из 1С;
  - важно обеспечивать, что эта дата действительно поддерживается и актуализируется внутри 1С (условие для проектирования).

### 5.3 Доставка и маршруты

- В 1С:
  - есть документ **«Маршрут»** (связан с реализацией, содержит дату отправки, адрес доставки);
  - есть **задание на перевозку** (оформляется логистом; содержит данные по машине, типу доставщика — своя/ТК, контакты водителя и др.);
  - фиксируются **точки прохождения маршрута** (по данным GPS).
- Для платформы:
  - критично получать:
    - факт создания маршрута и плановую дату;
    - данные о машине и водителе (для вывода в ЛК и уведомлений);
  - подробный трекинг по точкам маршрута в ЛК не обязателен — достаточно статуса «отправлен» и плановой даты/контактов водителя.

### 5.4 Порог бесплатной доставки (120 тыс.)

- В 1С **нет жёстко зашитого порога 120 тыс.**:
  - это бизнес-правило, зависящее от политики компании и может меняться (сумма выше/ниже).
- Рассмотрено два подхода:
  - хранить порог в 1С и передавать его в платформу;
  - **хранить порог в админке платформы** как общую настройку.
- В интервью склоняются к **варианту с админкой**:
  - порог — часть конфигурации сайта;
  - при изменениях достаточно правки в админке (без доработок обмена и 1С).

## 6. Онбординг клиентов и связь с 1С

### 6.1 Заявка «Стать клиентом» и создание контрагента

- Модель, к которой стороны склоняются:
  - клиент подаёт заявку на платформе (форма «Стать клиентом»);
  - платформа **не создаёт автоматически** контрагента в 1С;
  - менеджер получает заявку, **проверяет данные**, и **вручную заводит** контрагента/договор/соглашение в 1С.
- Причины отказа от автоматического создания контрагентов:
  - необходима верификация данных человеком;
  - риск ошибок/дубликатов в 1С;
  - наличие разных сценариев для сетей, ИП и т.п.

### 6.2 Идентификация и GUID

- В 1С:
  - у каждой сущности (контрагент, договор, заказ, документ) есть свой **GUID**;
  - для цен есть отдельные регистры (для них GUID номенклатуры используется как входной ключ).
- Для **сопоставления заявки с контрагентом**:
  - платформа может **генерировать свой GUID** заявки/клиента;
  - менеджер после создания/поиска контрагента в 1С **вносит этот GUID** в карточку контрагента/связанную сущность;
  - далее:
    - платформа и 1С обмениваются этим GUID как ключом соответствия;
    - email не используется как единственный ключ (он может отличаться, быть опечатанным или не совпадать с записями в 1С).
- **ИНН/КПП**:
  - используются при заведении и проверке контрагента;
  - для сетей (один ИНН, несколько КПП) нужно учитывать связку **ИНН + КПП**;
  - но в архитектуре интеграции основной ключ — **GUID**, а ИНН/КПП — атрибуты.

### 6.3 Роли на платформе и мастер-аккаунт

- На платформе предполагается, что:
  - **мастер-аккаунт** (директор/ответственный) регистрируется по email;
  - дальше в ЛК может **создавать под-пользователей** (закупщик, бухгалтер и др.);
  - в 1С email может:
    - совпадать с email в платформе,
    - или отличаться (например, корпоративный vs личный директора).
- Поэтому:
  - **связывание** ЛК ↔ 1С должно опираться на GUID контрагента/договора, а не только на email.

## 7. Интеграции, которые можно использовать как «донорские»

- У компании уже есть интеграции:
  - с **маркетплейсами** через купленное решение (код частично закрыт, недоступен для правок);
  - с **другой базой 1С** (складская система учёта) через **Apache** и обмен (HTTP).
- Важная находка:
  - есть расширение/модуль интеграции с **Яндекс.Маркетом**:
    - в нём есть **открытый код** по работе с API (обращения, асинхронные вызовы, методы);
    - его **реально можно использовать как основу/донор** для проектирования API‑обмена с новой платформой.
- Вывод:
  - стоит:
    - посмотреть код модуля Яндекс.Маркет;
    - использовать его как пример реализации HTTP‑сервисов, форматов и обработки ошибок;
    - адаптировать под нужды новой платформы (другая схема данных, свои методы).

## 8. Тестовый контур 1С и доступ разработчиков

- **Тестовая 1С есть**:
  - по сути это **копия боевой базы**, весом **>1 ТБ**;
  - переносить её куда-то к исполнителю нецелесообразно.
- План по доступу:
  - заказчик предоставляет **VPN** и настройки подключения;
  - настраивается отдельный **системный пользователь** для платформы;
  - команда заказчика (1С‑разработчики) помогает:
    - подключиться;
    - разобраться с конфигурацией;
    - показать, где лежат нужные справочники, заказы, документы.
- Это даёт возможность:
  - начинать проектировать и **тестировать API обмена до запуска фронта**;
  - параллельно:
    - фронт/дизайн — рисуются;
    - бэкенд — поднимает инфраструктуру и первую версию Swagger по спецификации.

## 9. Предварительные выводы и договорённости

1. **1С — источник истины**:
   - по товарам, ценам, контрагентам, заказам, статусам, документам и правилам доставки.
2. **EDO и «Честный знак»**:
   - сейчас — смешанный режим (бумага + ЭДО);
   - в перспективе все обязаны будут перейти к ЭДО; интеграцию стоит предусмотреть, но не делать «в лоб» в MVP.
3. **Механизм обмена**:
   - гибрид: расписание (фоновые данные) + события (статусы, ключевые документы);
   - инициаторами могут быть как платформа, так и 1С; шина/очередь (RabbitMQ и др.) — желательный слой для надёжности и масштабирования.
4. **Цены и скидки**:
   - платформа не хранит полностью пересчитанные уникальные прайсы по каждому контрагенту;
   - платформа получает из 1С **виды цен** (включая розничный для витрины) и **процент персональной скидки** по контрагенту/соглашению;
   - персональная скидка клиента **применяется в корзине** к базовым ценам по виду цены; разовые ручные скидки менеджеров остаются только в 1С.
5. **Порог 120 тыс.**:
   - не хранится в 1С в явном виде;
   - разумно вынести его в настройки платформы (админку), а не в интеграцию.
6. **Онбординг**:
   - автоматического создания контрагентов из платформы не будет;
   - менеджер верифицирует данные и заводит контрагента вручную;
   - для связки используется GUID + ИНН/КПП, а не только email.
7. **Тестовый контур 1С**:
   - есть и будет использоваться для разработки;
   - доступ через VPN и выделенного пользователя.
8. **Донорский модуль**:
   - модуль интеграции с Яндекс.Маркетом можно использовать как пример реализации HTTP‑сервисов и шаблон кода.
