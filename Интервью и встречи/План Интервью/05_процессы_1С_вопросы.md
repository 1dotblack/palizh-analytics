# 5. Процессы в 1С — вопросы к интервью

**Тема (из полного списка):** Что уже автоматизировано в 1С, что вручную. Структура данных: товары, цены, контрагенты, заказы, счета, правила доставки. Способы и архитектура обмена с платформой (API, шина). Список документов и данных, получаемых из 1С.

**Цель сессии:** оценить готовность 1С к интеграции, объём доработок, механизм обмена (API/расписание/вебхуки/шина), наличие тестового контура и базовый технический контракт обмена (авторизация, идентификаторы, ошибки).

**Участники (уточнить):** 1С (Стас Романов), при необходимости маркетинг по контенту.

---

## Контекст для интервью

- **ЛК менеджера в системе нет.** Оперативная переписка с клиентом — **внешние каналы** (в т.ч. **чат-виджет стороннего сервиса**, почта); структурированные заявки с портала — **формы** с email-маршрутизацией. Все данные по заказу и документы клиент получает через платформу (ЛК), а **источник этих данных — 1С**.
- **Вся документация по заказу** (счета, УПД, ТТН, паспорта качества, акты сверки и т.д.) планируется **получать из 1С** — платформа запрашивает или подписывается на обновления, 1С является источником истины.
- **Обмен с платформой предполагается по API** — важно зафиксировать: по расписанию или по событиям (вебхуки), кто инициирует (платформа вызывает 1С или общая шина сообщений), есть ли тестовая 1С и насколько она готова к интеграции.

## 1. Процессы и данные в 1С

- Какие процессы в 1С уже автоматизированы (заказы, счета, отгрузки, контрагенты)? Что делается вручную вне 1С?
- Как устроены в 1С: номенклатура, цены, остатки, контрагенты, заказы, счета, правила доставки? Есть ли уже API или выгрузки, которыми пользуетесь (маркетплейсы, другие сайты)?
- Кто отвечает за качество и актуальность данных для сайта: фото, описания, категории, свойства? Где это хранится и кто правит?
- Что потребуется **доработать в 1С для платформы** (новые выгрузки, приём заказов с платформы, отдача статусов и документов)?
- Есть ли ограничения по частоте выгрузки или по объёму данных (номенклатура, цены, контрагенты)?

### 1.1 Способ формирования заказа на платформе и приём в 1С

На витрине планируются **два контура набора позиций**: выбор в **каталоге** (корзина) и сценарий **шаблон электронной таблицы** — клиент скачивает файл, заполняет, загружает обратно. Нужно зафиксировать, как это **сводится к заказу в 1С**.

- Сегодня или в планах: есть ли в 1С **приём заказа из файла** (Excel/CSV) — вручную оператором, обработкой, **внешней обработкой загрузки**, отдельным **бизнес-процессом (БП)** или только через документ «Заказ клиента» с ручным вводом строк?
- Если таблица — **какой бизнес-процесс в 1С** (или цепочка документов) её обрабатывает: создание заказа сразу, черновик на проверку, задача менеджеру, согласование?
- **Сопоставление номенклатуры:** при загрузке таблицы — по чему матчится позиция (код, артикул, GUID номенклатуры, «номенклатура контрагента»)? Где в 1С правда при расхождении с файлом?
- **Корзина из каталога vs файл:** должен ли в 1С приходить **единый контракт** (например, тот же состав заказа в JSON), или для файлового канала предусмотрен **отдельный метод/очередь** и иная идемпотентность?
- Ограничения по **объёму** (строк в файле), **формату** шаблона, **валидации** до создания заказа: что делает 1С, что — платформа до передачи?

## 2. Механизм обмена: API, расписание, вебхуки, шина

- **Как будет проходить обмен?**
  - По **расписанию** (регламентные выгрузки/импорт в платформу) или по **событиям** (вебхуки из 1С при изменении заказа, появлении документа и т.д.)?
  - Платформа **сама обращается в 1С** (REST/HTTP API 1С) или планируется **общая шина данных** (например, RabbitMQ, Kafka), куда 1С публикует события, а платформа подписывается?
- Какая **конфигурация/редакция 1С** используется (УТ, ERP, свои доработки)? Есть ли уже HTTP-сервисы (REST/веб-сервисы) или обмен только файлами в папку?
- Если обмен идёт по API:
  - какой **формат** используется (JSON, XML, смешанный вариант);
  - как устроена **авторизация** (логин/пароль, токен, сертификат, VPN/white list);
  - есть ли уже **документация API** / описание методов / тестовая коллекция запросов.
- Ограничения по:
  - **частоте** обмена (периодичность регламентных заданий, лимиты на вызовы API);
  - **объёму** выгрузок (размер номенклатуры, количество контрагентов);
  - **времени отклика** при онлайн-запросах (получение документа по запросу из ЛК).
- Как в 1С хранятся и обновляются **виды цен**, **индивидуальные соглашения**, **номенклатура контрагента**? Есть ли единый регламент выгрузки/доступа к этим данным?
- Какие **внешние идентификаторы** будут использоваться в обмене: GUID контрагента, договора, соглашения, заказа, документа? Что будет ключом сопоставления на стороне платформы? Для онбординга текущее рабочее допущение уже принято: `GUID заявки` и флаг `активность` хранятся в карточке контрагента; нужно уточнить техническую реализацию этого правила в 1С.
- Как 1С должна обрабатывать **повторную отправку** одного и того же заказа/запроса: есть ли механизм идемпотентности, журнал обмена, повтор запросов при ошибке?
- Где и как будут видны **ошибки интеграции**: журнал регистрации 1С, почтовые уведомления, мониторинг, ручной разбор?

## 3. Тестовая 1С и готовность к интеграции

- **Есть ли тестовая (контурная) 1С** для разработки и приёмочного тестирования интеграции?
- **Степень готовности** тестовой 1С к работе с нами: те же конфигурация и доработки, что в проде? Доступны ли тестовые данные (номенклатура, контрагенты, заказы) и возможность «дергать» API или забирать выгрузки?
- Кто со стороны заказчика отвечает за **выдачу доступа** к тестовой 1С и за **согласование форматов** обмена (ответственный с 1С)?

## 4. Документы и данные: что планируется получать из 1С

Список для согласования с 1С: что будет отдаваться платформе (по запросу или по подписке), в каком формате и с какой периодичностью.

### 4.1 Документы по заказу и отгрузке (для отображения в ЛК)

| № | Документ/данные | Назначение на платформе | Формат/способ получения (уточнить с 1С) |
|---|------------------|-------------------------|------------------------------------------|
| 1 | **Счёт на оплату** | Просмотр/скачивание в ЛК, отправка клиенту по email | Печатная форма (PDF), по запросу или по событию «счёт сформирован» |
| 2 | **УПД** (универсальный передаточный документ) | Выдача клиенту в ЛК после отгрузки | По запросу из ЛК → 1С отдаёт документ (без хранения на платформе) |
| 3 | **ТТН** (товарно-транспортная накладная) | По запросу в ЛК | Печатная форма 1С |
| 4 | **Паспорт качества** | По заказу/партии в ЛК | Формируется в 1С; выдача по запросу или признаку контрагента |
| 5 | **Акт сверки** | Формирование по заявке из ЛК; выдача клиенту | Запрос из ЛК → формирование в 1С → выдача (email/ЭДО/скачивание в ЛК) |

### 4.2 Справочники и коммерческие данные

| № | Данные | Назначение на платформе |
|---|--------|-------------------------|
| 6 | Номенклатура, дерево групп, свойства | Каталог витрины (ЧТЗ 06) |
| 7 | Цены (виды цен), персональные условия по контрагенту | Отображение цен в каталоге/корзине/ЛК |
| 8 | Остатки, сроки производства | Наличие и «ожидаемая дата» в карточке и корзине |
| 9 | Контрагенты, договор, соглашение | Профиль компании в ЛК (реквизиты, лимиты, отсрочки, бонусы) |
| 10 | Правила доставки (пороги, направления) | Корзина, оформление заказа (ЧТЗ 03) |
| 11 | Номенклатура контрагента | Поиск и привязка «своё название ↔ название у клиента» (ЧТЗ 11) |

### 4.3 Заказы и статусы

| № | Данные | Назначение на платформе |
|---|--------|-------------------------|
| 12 | Заказы клиента (создание из платформы) | Приём заказа из ЛК в 1С (документ «Заказ клиента») |
| 13 | Статусы заказа (6 целевых статусов ЛК) | История заказов в ЛК; привязка к событиям/документам 1С (реализация, расходник, маршрут и т.д.) |
| 14 | Данные по доставке: маршрут, задание на перевозку, трек-номер ТК, контакты водителя | Карточка заказа в ЛК (трек, «своя машина» — водитель) |

### 4.4 Технические аспекты обмена

| № | Данные / объект | Что уточнить у 1С |
|---|------------------|-------------------|
| 15 | Внешние идентификаторы сущностей | Какие GUID/ID используются для контрагентов, договоров, заказов, документов; что платформа должна хранить как reference |
| 16 | Форматы и транспорт | JSON/XML, синхронный API, асинхронная очередь, файловый обмен как fallback |
| 17 | Ошибки и повторные запросы | Как выглядит ошибка 1С, как повторяем запрос, как избегаем дублей |

### 4.5 Дополнительно (при необходимости из 1С или из внешних хранилищ)

- TDS/MSDS, паспорт безопасности, СГР — если эти документы должны появляться в ЛК, их нужно предварительно загрузить/привязать в 1С, чтобы платформа получала их из 1С как из источника выдачи. Если документ не заводится в 1С, он остаётся вне автоматизированного контура ЛК и выдаётся менеджером вручную.
- Печатные формы 1С: способ выдачи (API отчёта, запрос по ID документа), кодировка, формат (PDF).

## 5. Что платформа передаёт в 1С

| Сущность | Назначение |
|----------|------------|
| Заказ клиента | Состав (позиции, количество), контрагент, адрес доставки, контакт → создание документа «Заказ клиента» в 1С; **варианты источника состава:** набор в каталоге (корзина) или **загрузка заполненной таблицы** — уточнить единый или разный путь в 1С и связанный **БП/обработку** |
| Заявка «стать клиентом» | Опционально: передача реквизитов для создания контрагента/договора или только уведомление менеджеру; рабочее допущение проекта — `GUID заявки` и флаг `активность` далее хранятся в карточке контрагента в 1С |
| Запрос акта сверки / инициация претензии | Передача заявки для процесса в 1С; если по претензии нужны документы в ЛК, они должны формироваться/загружаться в 1С |

## 6. Обновления по итогам саммари

*После обработки каждого саммари — дополнять: новые вопросы, что уточнить в следующих интервью.*

- **2026-03-13** (интервью 5 — 1С, обмен данными, цены): зафиксирована **версия 1С ERP 2.5.22**, наличие **тестового контура** и готовность использовать **гибридный механизм обмена**: часть данных по расписанию (регламентные задания/выгрузки), часть — по событиям/через шину сообщений. Подтверждён базовый список того, что платформа получает из 1С: документы по заказу (счёт, УПД, ТТН, акт сверки, паспорт качества и др.), справочники (номенклатура, дерево групп, свойства), коммерческие данные (виды цен, соглашения, персональные скидки, правила доставки), остатки и сроки производства, контрагенты/договоры/соглашения, заказы и статусы, данные по доставке (маршруты, трек-номера, задания на перевозку). Отдельно зафиксирована **модель цен и скидок**: платформа получает из 1С **виды цен** и **персональную скидку (%)** по контрагенту/соглашению и **применяет скидку в корзине на своей стороне**; от модели с уникальными пересчитанными ценами под каждого контрагента отказались; для **гостей** цены на витрине **не отображаются**, при этом из `1С` по товару может приходить ссылка на маркетплейс для перехода из карточки товара. Уточнять на следующих шагах: форматы и API для отдельных документов/отчётов, детальное поведение при ошибках и повторах (идемпотентность), финальный выбор транспорта (чистый API vs шина + fallback на файловый обмен), а также контракт поля / атрибута ссылки на маркетплейс.
