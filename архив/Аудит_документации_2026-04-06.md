# Отчёт-аудит документации проекта Palizh

**Дата:** 2026-04-06  
**Роль:** агент-аудитор (целостность ЧТЗ, техчасти, Инфарха, реестров).  
**Примечание:** отчёт не меняет исходные артефакты; ниже — рекомендации для аналитика/дизайнера.

---

## 1. Краткое видение проекта

Проект — **B2B-платформа** для клиентов Palizh с договором: **витрина и каталог** (в т.ч. без цен для гостя), **ЛК** с заказами, документами из 1С, доставкой и уведомлениями, **онбординг** через заявку «Стать клиентом», **интеграция с 1С** как источник истины по номенклатуре, ценам, заказам и документам. Дополнительно описаны **претензии**, **обучение** (MVP: контент в админке, заявки на email, история в ЛК), **админка** и **НФТ**. Документация в целом **согласована по основным контурам**, но есть **расхождения между историческими саммари/планом интервью и актуальными ЧТЗ**, а также **лаки в техконтракте и реестре открытых вопросов** относительно последних решений.

## 3. Проверка технической части

| Направление | Оценка | Замечания |
|-------------|--------|-----------|
| `openapi_mvp.yaml` | Нужно дополнить | Появились контуры **обучения** (`/public/training-programs`, `/training-applications`, `/me/training-applications`) и **пользователи компании** (`/me/company/users` …). В ЧТЗ 14 пути не обязаны совпадать дословно, но в `Маппинг_API_1С.md` / `Интеграция_1С.md` стоит добавить явную пометку «не 1С» для обучения и приглашений. |
| Интеграция 1С | ОК / уточнить | Соответствует гибридному обмену из саммари; детали по объектам — всё ещё открытые вопросы реестра. |
| Архитектура / НФТ | Нужно дополнить | Должны опираться на закрытие №50 в ЧТЗ 13. |

**Бизнес-логика в техдоках:** в целом уместна (контракты, жизненный цикл заказа); новые блоки лучше дублировать **одной строкой-ссылкой** в соответствующее ЧТЗ (07, 14), чтобы не плодить вторую правду.

**Недостаёт технически:**

- Явные правила для **приглашения под-пользователя**: TTL токена, идемпотентность повторной отправки, смена `email` до активации, запрет `PATCH status` из нелогичных переходов (например `invited` → без accept).
- Для **заявок на обучение**: идентификатор для ссылки в письме менеджеру ↔ экран в админке (уже в ЧТЗ 12 на уровне требования; в OpenAPI нет `/admin/training-applications/{id}` — при необходимости добавить в следующей итерации контракта).

## 6. Рекомендации по структуре артефактов

1. **Оглавление ЧТЗ** — в абзаце про сквозную зависимость от ЧТЗ 09 добавить **ЧТЗ 14** (обучение не из 1С в MVP, но влияет на ЛК, админку, уведомления).
2. **План интервью / Темы** — пометить устаревшие выводы по обучению (корзина 1С, B2C-скрытие колористики) или добавить ссылку «актуально см. ЧТЗ 14».
3. **Один мастер-документ по чату** — либо расширить ЧТЗ 10, либо ЧТЗ 12, со ссылками из остальных.
4. **Синхронизация статусов** заявки на обучение: в ЧТЗ — «Отправлено», в API — `sent`; в приглашениях — русские названия vs `invited` — завести таблицу в `Глоссарий.md` или в `Сводка_онбординг_и_идентификаторы.md`.

**Явные потенциальные противоречия (что принять за истину):**

- **Обучение:** истина — **ЧТЗ 14 + Инфарх + Оглавление** (публичная витрина, заявка только авторизованным, история в ЛК, без 1С в MVP). Старые саммари, где «колористика только B2B» и «экскурсии позже», считать **историческими**, не опорными для спека.
- **Порог доставки:** до явного решения — зафиксировать приоритет: **настройка платформы (ЧТЗ 12)** vs **правила из 1С**; сейчас в ЧТЗ 12 указано, что порог бесплатной доставки — параметр платформы для MVP.
