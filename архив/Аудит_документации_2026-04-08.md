# Отчёт-аудит документации проекта Palizh

**Дата:** 2026-04-08  
**Роль:** агент-аудитор (по `.cursor/rules/agent-auditor.mdc`)  
**Охват:** ЧТЗ 01–14, 13 НФТ, Оглавление, Глоссарий, матрицы; `Техническая часть/`; Инфарх (видение, состав страниц, ИА — выборочно); реестры вопросов и документов; правило `project-analyst.mdc`.

Документы **не изменялись** в ходе аудита — только рекомендации для аналитика и команды.

## 1. Краткое видение проекта

Проект — **B2B-платформа** Palizh: **витрина и каталог** (цены для гостя скрыты), **ЛК** с ролями, **оформление заказа**, **документы из 1С** (счёт, УПД, ТТН, паспорт качества, акт сверки и др.), **доставка** (порог и подсказки в админке; операционные данные — из 1С по JSON API), **онбординг** «стать клиентом», **претензии (MVP)**, **обучение (MVP — заявка на email)**, **админка**, **интеграция с 1С** как сквозной контур. Документация в целом **согласована по контурам**; основные разрывы — **детализация контракта 1С ↔ платформа** (JSON, идентификаторы, файлы), **синхронизация формулировок** после уточнений по реестру документов и доставке, **устаревшие фразы** в реестре открытых вопросов и **лёгкое расхождение** Инфарх/ЧТЗ по «правилам доставки» vs админка.

## 4. Реестр открытых вопросов

- **Де-факто устаревшие формулировки:**
  - В конце файла (блок про реестр документов): «образцы … для проектирования **обмена с 1С**» — по текущему [Реестр_документов_для_проектирования.md](Интервью%20и%20встречи/Реестр_документов_для_проектирования.md) реестр = **файловые образцы для ЛК**, обмен — **ЧТЗ 09 / JSON API**.
  - Раздел «Осталось закрыть до MVP», пункт **№25**: сузить до **ТТН + протокол разногласий** (остальное по реестру получено или вынесено из реестра).

- **Потенциальный конфликт с ЧТЗ 02:**
  - Закрытый **№22** говорит про документы, «связанные с **заказом**» для TDS/СГР; в **ЧТЗ 02 §4.0** номенклатурные документы в MVP — прежде всего раздел «Документы» ЛК. **Рекомендация:** в реестре вопросов добавить сноску или уточнить формулировку №22, чтобы не читалось как противоречие матрице документов.

- **№31 (ПЭО):** закрыт как «фокус на своей доставке»; **ЧТЗ 03** уже уточнён (ПЭО-документ не для платформы, порог в админке). Дополнительных действий не требуется, кроме выравнивания **Инфарх** (см. §6).

- **Критичные для старта разработки:** №18, 30, 34, 36, 37, 40, 47, 48, 49, 50 — уже помечены MVP; дублей между ними нет, но **№36 и ЧТЗ 02 §4.0** логически одна тема (какие SKU-документы в 1С для ЛК).

## 6. Рекомендации по структуре артефактов

1. **`ЧТЗ/Оглавление.md`:** в абзаце про сквозную зависимость ЧТЗ 09 добавить **13 (НФТ)** и **14 (обучение)** там, где они реально зависят от обмена/уведомлений; в планируемых блоках уточнить ТК (**СДЭК не планируется** по ЧТЗ 03 — не вводить в заблуждение списком «СДЭК, ПЭК»).
2. **`Инфарх/видение-и-границы.md`**, пункт про интеграции: сейчас «**правила доставки**» из 1С; по **ЧТЗ 03** порог/подсказки — **админка**, из 1С — **операционные delivery-данные**. Сформулировать так, чтобы не казалось, что порог приезжает из 1С.
3. **Один источник по НФТ:** ЧТЗ 13 ↔ `Техническая часть/Нефункциональные_требования.md`.
4. **`uscases/`** и **`Аудит_документации_2026-04-06.md`:** при желании — одна ссылка из README или оглавления аналитика на актуальный аудит (**этот файл**), старый аудит пометить как архивный или сократить дубли с новым.
